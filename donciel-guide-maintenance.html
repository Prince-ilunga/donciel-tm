Elements a Conserver

1.1 Comptes et Acces

Les comptes suivants sont essentiels au fonctionnement de l'application DONCIEL TM. Conservez ces informations en lieu sur et assurez-vous que les acces restent valides.

Service Identifiant / URL Role Priorite
GitHub Prince-ilunga/donciel-tm Depot de code source Critique
Vercel https://donciel-trading.vercel.app Hebergement & deploiement Critique
Neon PostgreSQL Connection string dans .env Base de donnees hebergee Critique
Cloudinary Cloud name, API key & secret dans .env Stockage medias (captures d'ecran, videos) Important
JWT / NextAuth NEXTAUTH_SECRET dans .env Securite d'authentification Critique
Attention Ne partagez jamais les cles d'API ou les secrets JWT publiquement. Utilisez toujours le tableau de bord Vercel pour configurer les variables d'environnement en production.

1.2 Variables d'Environnement (.env)

Le fichier .env contient toutes les configurations sensibles de l'application. Ces variables doivent etre definies a la fois localement et sur Vercel.

Variable Description Exemple Obligatoire
DATABASE_URL URL de connexion Neon PostgreSQL postgresql://user:pass@ep-xxx.neon.tech/db Oui
NEXTAUTH_SECRET Secret pour le chiffrement JWT NextAuth votre-secret-aleatoire Oui
CLOUDINARY_CLOUD_NAME Nom du cloud Cloudinary donciel-tm Prod
CLOUDINARY_API_KEY Cle API Cloudinary 123456789012345 Prod
CLOUDINARY_API_SECRET Secret API Cloudinary abcdefg... Prod
Note En developpement local, DATABASE_URL peut pointer vers un fichier SQLite local (ex: file:/home/z/my-project/db/custom.db). En production sur Vercel, elle doit pointer vers l'instance Neon PostgreSQL.

1.7 Modeles de Base de Donnees (Prisma)

Le schema Prisma definit les modeles suivants dans la base PostgreSQL :

Modele Description Champs cles
User Utilisateurs de l'application email, password, name, role (admin/user), status (pending/approved/rejected), language (fr/en)
Trade Enregistrements de trades date, pair, direction, session, marketCondition, timeframe, setup, structure, entryModel, entryPrice, stopLoss, takeProfit, exitPrice, rr, pnl, result, emotions, confluence, mistakes, lessons
Screenshot Captures d'ecran des trades type (analysis/entry/exit), url
Video Videos educatives category (STRUCTURE/BIAIS/ZONES/MODELS/SETUPS), title, description, url
Note Notes de preparation type (DAY/WEEK/MONTH), title, content, date
CustomPair Paires de devises personnalisees name
UploadSession Sessions d'upload chunked uploadId, filename, totalSize, totalChunks, category, receivedChunks
Relations Un User possede plusieurs Trade, Note, CustomPair et Video. Un Trade possede plusieurs Screenshot. La suppression d'un utilisateur entraine la suppression en cascade de toutes ses donnees associees.

PARTIE 2

Mode d'Emploi — Mise a Jour

2.1 Mise a jour du code source

Pour recuperer les dernieres modifications du depot GitHub :

$ git pull origin main
Bonne pratique Avant de tirer les modifications, assurez-vous d'avoir commite vos propres changements avec git add . et git commit -m "message" pour eviter les conflits.

2.2 Mise a jour des dependances

Apres avoir mis a jour le code source, installez les eventuelles nouvelles dependances :

$ bun install

Cela mettra a jour le dossier node_modules selon le fichier package.json et bun.lock.

Scripts disponibles

Commande Description
bun run devLance le serveur de developpement sur le port 3000
bun run buildGenere le client Prisma puis compile le projet
bun run startLance le serveur de production
bun run lintVerifie la qualite du code avec ESLint

2.3 Mise a jour de la base de donnees

Si le schema Prisma (prisma/schema.prisma) a ete modifie, vous devez synchroniser la base de donnees :

$ bun run db:push # ou equivalemment: bunx prisma db push

Commandes Prisma utiles

Commande Description
bun run db:pushPousse le schema vers la DB (sans migration)
bun run db:generateRegenere le client Prisma
bun run db:migrateCree et applique une migration
bun run db:resetReinitialise la base de donnees (dangereux !)
Danger La commande db:reset supprime toutes les donnees de la base. Ne l'utilisez qu'en cas de reinitialisation complete. Effectuez toujours une sauvegarde avant.

2.4 Deploiement sur Vercel

Le deploiement est automatique : chaque push sur la branche main declenche un nouveau deploiement sur Vercel.

$ git push origin main
  1. Effectuez vos modifications en local et testez avec bun run dev
  2. Commitez vos changements : git add . puis git commit -m "description"
  3. Poussez sur GitHub : git push origin main
  4. Vercel detecte le push et lance le build automatiquement
  5. Une fois le build termine, le site est mis a jour sur donciel-trading.vercel.app
Zero-downtime Vercel deploie les nouvelles versions sans interruption de service. L'ancienne version reste active jusqu'a ce que la nouvelle soit prete.

2.5 Verification apres deploiement

Apres chaque deploiement, verifiez les points suivants :

  1. Visitez https://donciel-trading.vercel.app et verifiez que la page d'accueil se charge
  2. Connectez-vous avec un compte existant et verifiez l'authentification
  3. Consultez le dashboard Vercel pour verifier les logs de build et d'execution
  4. Testez les fonctionnalites cles : ajout de trade, journal, graphiques
  5. Verifiez que l'upload de captures d'ecran fonctionne (Cloudinary)
Dashboard Vercel Accedez au dashboard a l'adresse https://vercel.com/dashboard pour voir les deploiements, les logs en temps reel et les erreurs eventuelles.

2.6 Sauvegarde de la base de donnees

Il est recommande d'effectuer des sauvegardes regulieres de la base de donnees Neon PostgreSQL.

Option 1 : Neon Console

  1. Connectez-vous a https://console.neon.tech
  2. Selectionnez votre projet
  3. Utilisez la fonctionnalite d'export ou de branch pour creer une sauvegarde

Option 2 : pg_dump en ligne de commande

# Sauvegarde complete de la base $ pg_dump "postgresql://user:pass@ep-xxx.neon.tech/dbname" > backup_$(date +%Y%m%d).sql # Restauration $ psql "postgresql://user:pass@ep-xxx.neon.tech/dbname" < backup_20260304.sql
Recommandation Effectuez une sauvegarde au moins une fois par semaine, et systematiquement avant toute modification du schema de base de donnees.

2.7 En cas de probleme

Guide de resolution des problemes courants :

Le site ne se charge pas

  • Verifiez le statut du deploiement dans le dashboard Vercel
  • Consultez les logs d'erreur dans la section Functions de Vercel
  • Verifiez que les variables d'environnement sont correctes sur Vercel

Erreur de base de donnees

  • Verifiez que DATABASE_URL est correct dans les variables Vercel
  • Assurez-vous que le schema Prisma est synchronise : bun run db:push
  • Verifiez que la base Neon n'est pas en pause ( Neon met en pause les bases inactives)

Les uploads ne fonctionnent pas

  • Verifiez les variables CLOUDINARY_CLOUD_NAME, CLOUDINARY_API_KEY, CLOUDINARY_API_SECRET
  • Testez les identifiants dans la console Cloudinary
  • Verifiez les limites de stockage de votre plan Cloudinary

Erreur d'authentification

  • Verifiez que NEXTAUTH_SECRET est defini sur Vercel
  • Regenerez le secret si necessaire et mettez a jour partout
  • Verifiez que la base de donnees contient bien les utilisateurs

Erreur de build Vercel

  • Lisez attentivement les logs de build dans le dashboard Vercel
  • Verifiez que bun run build fonctionne en local sans erreur
  • Assurez-vous que prisma generate est execute avant le build (il est dans le script build)
Ressources utiles
  • Documentation Vercel : https://vercel.com/docs
  • Documentation Prisma : https://www.prisma.io/docs
  • Documentation Neon : https://neon.tech/docs
  • Documentation Cloudinary : https://cloudinary.com/documentation

2.8 Checklist de Maintenance

Utilisez cette checklist pour chaque intervention de maintenance :

# Tache Commande / Action
1 Sauvegarder la base de donnees pg_dump ou Neon Console
2 Recuperer le dernier code git pull origin main
3 Installer les dependances bun install
4 Synchroniser la base de donnees bun run db:push (si schema modifie)
5 Tester en local bun run dev puis tests manuels
6 Deployer en production git push origin main
7 Verifier le deploiement Visiter donciel-trading.vercel.app
8 Verifier les logs Vercel Dashboard Vercel > Logs
9 Confirmer les fonctionnalites Tests manuels des fonctionnalites cles
